## 三、AI 协作契约

### 团队名称：404 Not Found
### 日期：2026.4.6

---

#### 第一条：AI 使用原则
- AI 是工具，工程师是责任主体
- 所有 AI 输出必须经过人工审查后才能合入项目

#### 第二条：AI 使用范围

**允许使用 AI 的场景：**
- 代码补全、自动补全和智能提示
- 生成单元测试用例和测试代码
- 代码重构建议和优化方案
- 文档生成（如 JSDoc、README 草稿）
- 调试辅助和错误诊断
- 学习新技术和查阅 API 文档

**禁止直接使用 AI 输出（需额外审查）的场景：**
- 核心业务逻辑代码（如支付、权限校验）
- 涉及安全和隐私的代码（如加密、认证）
- 数据库 Schema 设计和关键 SQL
- 第三方库选型和技术决策

**完全禁止使用 AI 的场景：**
- 考试或作业中明确禁止 AI 的部分
- 涉及学术诚信的原创性内容
- 处理真实用户隐私数据

#### 第三条：过程记录要求
- 所有与 AI 的交互 Prompt 必须保留日志
- Git commit message 必须标注 `[AI-assisted]` 或 `[Human-written]`
- 每阶段填写《AI 协作反思日志》

#### 第四条：代码合入规则
- AI 生成的代码必须通过至少一名团队成员的 Code Review
- 每位成员每个编码阶段至少有 2 个完全手写的核心函数

#### 第五条：违约处理
- 首次违约：团队内部警告，记录备案
- 重复违约：在团队会议上公开检讨
- 严重违约（如未审查的 AI 代码导致线上事故）：取消该成员本阶段评优资格

## 四、团队协作约定

### 会议频率
- 每周一次线下站会（周三 16:00，时长 30 分钟）
- 每日线上同步（QQ群，简要汇报进度和阻塞）

### 沟通方式
- 日常沟通：qq群
- 技术讨论：GitHub Issues / Discussions
- 文档协作：飞书文档

### 代码提交规范
- 分支策略：`main`（受保护）→ `dev`（集成）→ `feature/*`（开发）
- Commit message 遵循 Conventional Commits 规范
- PR 必须经过至少 1 人 Review 才能合并

### 文档规范
- 所有技术决策必须写 ADR（架构决策记录）
- API 变更必须同步更新接口文档
- 每阶段结束更新《AI 协作反思日志》
